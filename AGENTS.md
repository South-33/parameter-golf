# Notes

- Collaboration fallback -> if an agent hits an environment limitation, harness issue, missing access, or any blocker it cannot safely resolve, it should ask the user for help instead of guessing or stalling -> keeps loops moving when the local setup is the bottleneck.
- Shared ideation backlog -> `ideas.md` is the living ranked idea backlog and experiment ledger; agents should update it when a new idea appears, when rankings change, and whenever a tried idea gets a result -> keeps the current strategy and evidence in one place.
- Complaint ledger -> `complain.md` is the place to capture recurring harness/process/infrastructure pain points that deserve later cleanup or prioritization -> prevents expensive friction from being forgotten between runs.
- Complaint follow-through -> once a complaint in `complain.md` is clearly real and recurring, agents should either fix it directly or explicitly consult the user about prioritizing it -> avoids building a passive graveyard of known friction.
- Research loop tracking -> `ideas.md` also owns the research/prompt-evolution log; when agents run web/arXiv/subagent research they should record the question asked, what signals were genuinely useful, what was noise, and how the next prompt should improve -> prevents repeating weak research passes and helps the harness get sharper over time.
- Agent posture -> stay goal-oriented and evidence-driven; preserve useful freedom for smart models instead of over-specifying tasks or assuming the current plan is final -> keeps future agents adaptive when new evidence beats the current intuition.
- Subagent default -> treat subagents as smaller, faster junior researchers; default to a mini model (for example `gpt-5.4-mini`) and keep main implementation, integration, and final judgment on the main thread unless parallelism clearly saves time -> avoids wasting overhead on work the main agent should just do directly.
- Subagent write scope -> subagents are for research, exploration, and bounded read-only scouting; project file edits, integration, and any real worktree changes should happen on the main thread unless the user explicitly wants delegated code changes -> prevents hidden file edits, merge conflicts, and harder-to-audit state.
- Local Python setup -> the repo is on `C:`, but the working Python environment is at `D:\venvs\parameter-golf`; prefer that venv for installs and runs because `C:` space is tight -> prevents repeated CUDA install failures from disk pressure.
- Windows local launch -> for single-GPU local loops prefer `python train_gpt.py` over `torchrun`; the Windows launcher path can trip over libuv/rendezvous issues that do not matter for the 1-GPU smoke workflow -> avoids wasting time on launcher-specific failures.
- Timed-out CUDA probes -> on this Windows 4060 setup, timed-out local Python evals can leave orphaned GPU worker processes behind; check `nvidia-smi` / `Get-Process python` and clean them before the next run if utilization looks stuck -> avoids poisoned timing and fake slowness from stale workers.
- Windows artifact writes -> repeated local runs can occasionally fail saving `final_model.pt` / `final_model.int8.ptz` with Windows file-lock error `1224`; if that happens, remove the output artifacts before rerunning and restore tracked artifacts afterward -> avoids losing a good probe to a save-only failure.
- Strict local determinism mode -> `STRICT_DETERMINISM=1` disables TF32/fused Adam and tightens CUDA determinism for diagnostics, but it materially changes speed and still did not make short 10-step scores reliably reproducible -> use it to debug harness drift, not as the default ranking mode.
- Eval-to-train cache hazard -> step-0 validation runs under `torch.inference_mode()`, so reusable tensor caches must not be populated there unless they are rebuilt outside inference mode; RoPE caching hit this and silently broke later training -> avoids fake "slow hangs" that are really eval/train cache bugs.
- Validation stride -> `EVAL_STRIDE_TOKENS<TRAIN_SEQ_LEN` switches `eval_val` into overlapping suffix scoring instead of plain full-sequence loss; compare scores only with the same stride setting -> avoids mixing sliding-window and full-context results.
- Doc-isolated eval callsite -> after changing `eval_val(..., bos_id)`, update every post-train / post-quant eval call too; the missed final int8 roundtrip call crashed only after successful export on Runpod -> avoids wasting cloud runs on a late signature mismatch.
- Short local proxy noise -> the 10-step stride-64 post-quant proxy can move noticeably across reruns even with the same nominal branch, so judge small changes against same-code controls rather than mixing results from distant code states -> avoids over-claiming tiny wins or regressions.
- Exporter probe hygiene -> for quant/export ideas, same-checkpoint exporter-only evals on a saved checkpoint are more trustworthy locally than short retraining reruns -> avoids mistaking training noise for exporter wins.
- Kurtosis branch caution -> `KURTOSIS_PENALTY=1e-5` looked good on one fixed-path rerun but collapsed badly on a later confirmation despite the same nominal setup -> treat it as unstable/noisy on the short local loop rather than a promoted default.
- Post-quant control tuning caveat -> late calibration steps on train data can help only when added after enough base training; they did not survive tighter `9/1` or `8/2` train/tune budget splits locally, so do not treat them as a free budget-respecting default -> avoids fake wins from hidden extra optimization.
- Git checkpoints -> make descriptive checkpoint commits after substantial progress and push when the result is worth preserving remotely -> keeps rollback points available during long experiment loops.
- Experiment file hygiene -> keep ad hoc experiment/result text snapshots under `experiments/` instead of the repo root; reserve `logs/` for run logs and keep the root focused on source/config files -> prevents the project root from turning into an unreadable dump of probes.
- Local dev environment -> primary machine is Windows 11 with an RTX 4060 8GB -> use it for CUDA smoke tests and iteration, not as a proxy for 8xH100 leaderboard performance.
- Submission scoring -> final metric is post-quant `final_int8_zlib_roundtrip_exact val_bpb`, not the pre-quant validation loss -> optimizing a model that breaks after int8+zlib can look good mid-run but still fail the leaderboard metric.
- Artifact accounting -> the 16,000,000-byte cap includes counted code plus compressed model bytes, and challenge submissions are expected to keep counted code in `train_gpt.py` while record PRs add a new folder under `records/` -> changing only root trainer code is not a valid final submission shape.
- Evaluation/data -> validation is tokenizer-agnostic BPB on the fixed first-50k-doc `fineweb_val_*` split; tokenizer changes are allowed but scrutinized and must preserve correct byte accounting -> tokenizer bugs can create invalid wins.
- Tokenizer branch caveat -> the local data helper accepts `sp4096`, but the current published Hugging Face manifest only exposes `sp1024`, so an SP-4096 branch requires local tokenizer/data re-export rather than a simple cached download -> avoids wasting time assuming the higher-vocab dataset is already hosted.
- Tokenizer artifact accounting -> Issue #43 suggests the tokenizer file itself may not count toward submission bytes, pending maintainer clarification; this makes tokenizer changes strategically stronger than our earlier informal byte model -> do not under-rank tokenizer ideas just because the `.model` file is larger.
- Tokenizer re-export cost -> the hosted `datasets/docs_selected.jsonl` needed for local tokenizer/data re-export is about `48.17 GB`; prefer `D:` for any tokenizer re-export workspace and treat this as a substantial download/storage decision, not a quick prep step -> avoids accidentally blowing local time or disk on the wrong drive.
- Tokenizer overnight path -> `data/tokenizer_specs_sp4096_only.json` limits the rebuild to just `sp4096`, which is the useful overnight branch here -> avoids wasting time and disk rebuilding extra tokenizer variants.
- Manual docs cache preload -> `data/download_hf_docs_and_tokenize.py` now reuses a pre-downloaded `docs_selected.jsonl` and sidecar already sitting in the chosen output root -> lets browser/download-manager fetches feed the rebuild directly.
- Runpod HF cache path -> on the `runpod/parameter-golf:latest` pod, Hugging Face downloads default to the small `/root` overlay and can fail even with a large `/workspace` volume; set `HF_HOME` / `HUGGINGFACE_HUB_CACHE` under `/workspace` before pulling `docs_selected.jsonl` -> avoids fake disk exhaustion on cloud rebuilds.
- Runpod cost discipline -> treat running GPU pods as active work only; stop them immediately after a result or blocker, preserve at least a few dollars for one final 8xH100-class submission shot, and prefer reusing migrated data over recreating pods -> avoids bleeding credits on idle cloud time.
- Runpod launch hygiene -> keep repeatable pod runs in repo scripts (for example `experiments/runpod_sp4096_smoke.sh`) and pull them onto the pod instead of embedding giant env/command strings in Windows SSH calls -> avoids losing results to quoting and line-ending bugs.
- Runpod smoke log path -> the meaningful trainer output on cloud smoke runs lands in `logs/<RUN_ID>.txt`; the outer `nohup`/driver log may stay empty or only print that redirected path -> tail the run-id log for progress and exact metrics.
- Runpod wrapper -> `experiments/runpod_launch_wrapper.sh` is now the canonical cloud launcher; it writes `.status/.done/.failed/.pid` next to the run-id logs and is much safer than ad hoc SSH command strings -> use it for future pod jobs by default.
- SP-4096 width tradeoff -> on the capped cloud smoke, `640/1664` beat `768/2048` (`2.8440` vs `2.8782` exact) while both were far under the byte cap -> spend extra SP-4096 headroom on higher-signal branches like tied-embedding protection before trying more plain width.
- SP-4096 FP16 tok-emb export -> keeping `tok_emb.weight` fp16 at `640/1664` raised total size from `6.12 MB` to `8.15 MB` and was slightly worse (`2.84423` vs `2.84401` exact) on the same capped cloud smoke -> do not prioritize exporter-only FP16 token embedding on this branch.
- SP-4096 seq2048 smoke -> `TRAIN_SEQ_LEN=2048` on the same `640/1664` branch was also slightly worse (`2.84642` exact vs `2.84401` at `1024`) on the capped cloud smoke while staying under `6.11 MB` counted size -> do not assume the leaderboard long-context signal transfers to this tiny 1x3090 smoke regime.
- SP-4096 fertility check -> first pod-side 5k-doc probe measured `sp1024` at `0.4158` tokens/byte and `sp4096` at `0.3067` (`0.7376x` ratio), which is strong enough to justify the full re-export -> treat tokenizer as a live branch, not just theory.
- Pod watcher quirk -> long remote jobs can be monitored from this harness with a blocking poller, but a pure file-count stall heuristic can false-trigger right as the job exits; trust the pod-side `.done`/manifest success markers over the stall guess -> avoids misclassifying finished exports as stuck.
- Validation doc boundaries -> current dataset export prepends `BOS` to every doc but does not append `EOS`, so doc-isolated eval must split on `BOS` and preserve whole docs instead of relying on EOS markers -> avoids silently scoring cross-doc context or truncating documents.
- Exporter direction -> row-centered int8 export is width-sensitive: it consistently helped the local `9/3 @ 1024` branch for ~36 KB extra compressed size, but slightly hurt the current best `9/3 @ 896` branch, while grouped per-row scales were mostly noise -> treat centered export as a recovery tool for wider models, not a blanket default.
- Tied embedding precision -> the tied input embedding / output head looks important enough to protect during both training and export; PR #10 adds the useful nuance that it should stay fp32-master during optimization, not only get special treatment at export -> treat tied-embedding precision as a full branch, not just a serializer tweak.
- Tied embedding local evidence -> exporter-side protection still has some signal, but the short 4060 proxy did not show a free win from `TIED_EMB_FP32_MASTER=1` during training -> do not overfit to the PR nuance without a better-quality eval.
- Activation-aware `v <-> proj` -> one short-loop pass improved at `alpha=0.5`, but a fresh matched confirmation rerun lost badly to weight-only `vproj` -> treat this branch as noisy/testing rather than promoted.
- Research direction -> recent research passes pointed toward rotation/incoherence and scale-reparameterization as promising families, but the fixed Hadamard exporter probe underperformed; exact weight-only `v <-> proj` equalization still has the cleanest signal so far, while activation-aware `vproj` and `relu^2` MLP equalization both need caution -> treat this as current evidence, not a locked roadmap.
- Leaderboard direction -> official records are currently dominated by doc-isolated sliding eval, longer context, lower LR/longer warmdown, and tied-embedding precision; exporter-only micro-tweaks look secondary by comparison -> bias new work toward eval/schedule/context before more quant nibbling.

## Evidence Ladder

- `8xH100 final exact` > `same-branch cloud exact final` > `same-checkpoint exporter-only exact` > `capped cloud smoke exact` > `local short proxy`.
- Promotion rule -> do not let a weaker rung overrule a stronger rung without a concrete reason; use lower rungs to filter bad ideas, not to claim wins.
- Comparison rule -> only compare results from the same rung unless the purpose is explicitly to decide whether a branch deserves promotion to a stronger rung.

## Main Goals

- Official challenge target -> minimize held-out loss on the fixed FineWeb dataset while staying within a strict `16 MB` artifact limit and a `10 minute` training budget on `8x H100s` -> this is the actual optimization problem from the OpenAI challenge page and repo.
- Real win condition -> lower the final post-roundtrip metric `final_int8_zlib_roundtrip_exact val_bpb` on a leaderboard-safe run -> local pre-quant improvements alone are not wins.
- Good wins -> architecture, quantization, tokenizer, or systems changes that improve final post-roundtrip `val_bpb`, preserve artifact viability, and plausibly scale to the real challenge budget -> these are the changes worth spending time on.
- Weak wins -> local-only speedups, tiny pre-quant gains that vanish after quantization, or tweaks that only help the Windows smoke loop -> do not confuse iteration convenience with challenge progress.
- Local loop purpose -> use the RTX 4060 8GB setup to reject bad ideas quickly and identify promising ones for later serious runs -> treat local numbers as directional, not authoritative.
